# KeyboardLock — Spec Review

**VERDICT: Not ready — needs revision.** Five blockers tied to the safety story (watchdog concurrency, menu-bar unlock bypass, hotkey-tail leak) and one cross-document contradiction (Sparkle vs. "no network calls"). The tap design is broadly sound; the failure paths around it are not.

## Findings

| ID | Severity | Section | Problem | Fix / Question |
| --- | --- | --- | --- | --- |
| REV-1 | **Blocker** | EDGE-1 / ARCH-4 | Watchdog mutates `LockStateMachine.state` from a background queue when main is hung. `state` is `@Published` on a `@MainActor ObservableObject`; setting it off-main is a concurrency violation and will trap (or silently corrupt) under Swift Concurrency checking. The spec hand-waves this as "the one exception." | Split state into two stores: an atomic `LockEnforcement` (owned by `LockController`, mutable from any thread) that drives tap installation, and a `@MainActor` mirror that the UI observes. The watchdog touches only `LockEnforcement` + tap teardown + IOPM release. The mirror catches up whenever main wakes. Do NOT mutate `@Published` off-main. |
| REV-2 | **Blocker** | UI-5 / FR-7 | The menu-bar menu exposes a plain "Unlock Keyboard" item with no hold/double-click safeguard. PRD FR-7 *requires* a safeguard against accidental unlock. Two stray mouse interactions (cloth on trackpad opens the menu, second drag selects "Unlock Keyboard") defeat the entire safety model. | Either (a) remove the Unlock item from the menu while locked and route to the floating panel's HoldButton, or (b) make the menu item itself require hold-to-confirm (NSMenu doesn't support this well — option (a) is cleaner). The PRD-compliant answer is (a). |
| REV-3 | **Blocker** | FSM-4 last sentence | Hotkey tail-of-chord handling is broken. The spec consumes the matched `keyDown`, dispatches unlock to main, and then says the matcher will also swallow `keyUp` "for 500 ms" — but the tap is torn down as part of the unlock transition, so the subsequent `keyUp` events for ⌃, ⌥, ⌘, L arrive with no tap installed and reach the frontmost app. With ⌃⌥⌘L, this is unlikely to cause damage; with a user-configured hotkey like ⌃⌥⌘S (Save) the `keyDown` is suppressed but the `keyUp` arriving after teardown can still trigger menu-item activation in some apps. More importantly the spec is internally contradictory. | Tear the tap down **after** a debounce: keep the tap installed for ~250 ms after `unlockRequestedByHotkey`, suppress all events during that window, then teardown. State machine adds an explicit `unlockingDrain` sub-state. |
| REV-4 | **Blocker** | BUILD-5 vs. PRD §6.5 / SPEC §1 intro | Direct contradiction. PRD §6.5: "No analytics, **no network calls in v1**." Spec §1 intro table: "Third-party deps: None required in v1." BUILD-5 then mandates Sparkle, which is a third-party dep that makes recurring network calls to fetch the appcast. Cannot all three be true. | Pick one: (a) defer Sparkle to v1.1 and ship "check for updates" as a manual mouse-driven button that opens the GitHub Releases page (zero background network), OR (b) get PM to amend PRD §6.5 to permit Sparkle's appcast fetch. Recommend (a) for v1 to keep the privacy claim load-bearing. |
| REV-5 | **Blocker** | TEST-M6 | The test "attach lldb and process interrupt for 6 s" cannot validate the watchdog. `process interrupt` halts **all threads** in the target, including the watchdog's DispatchSourceTimer thread and the tap thread. When you resume, the watchdog's wall-clock check (`now - lastHeartbeatAt > 5.0`) using `ContinuousClock` *will* see the gap, but only because ContinuousClock advances during the pause — which is true on Apple platforms, but is the *opposite* assumption from a hang where threads are running but starved. The test passes for the wrong reason and gives false confidence in the production code path. | Replace with: (a) a unit test that injects a `Clock` into the watchdog and a fake heartbeat source, and (b) a manual test that runs a CPU-bound infinite loop on the main thread (`while true {}` triggered via a debug menu) so the main queue is genuinely starved while other threads run. The TEST-U5 design already implies (a) but the production watchdog lacks the injection seam. |
| REV-6 | Major | TAP-5 | Re-arm after `kCGEventTapDisabledByTimeout` leaves an unbounded gap during which every keystroke reaches the frontmost app. Spec says "count up to 3 then give up" but does not bound the per-disable gap, does not guarantee any cap on leaked events per session, and does not describe what the floating panel UI does during the gap. For a safety-critical app the contract should be "zero keys leak" not "occasional bursts." | Specify: on `tapDisabledByTimeout`, (1) immediately set the floating panel to a "Re-arming…" visual; (2) re-enable on the same callback invocation (Apple's recommendation — the callback is on the tap thread); (3) if re-enable returns failure, transition to `unlockedReady` with an explicit "lock interrupted" banner. Also tighten the callback hot path with a budget (no allocations, no Swift String work) so timeouts are rare in practice. |
| REV-7 | Major | UI-9 / PRD FR-9a | FR-9a says ⌥⌘Esc must remain reachable as a fallback. The spec's tap at `kCGHIDEventTap` will consume ⌥⌘Esc before the OS sees it, so the macOS Force Quit dialog is **not** reachable from the keyboard while locked. The spec acknowledges Force Quit via the mouse-driven Apple menu (good) but does not state that ⌥⌘Esc is unreachable. PRD FR-9a is therefore unsatisfied as written. | Two options: (a) add ⌥⌘Esc to the matcher as a permanently-allowed chord that passes through (same allowlist mechanism as the unlock hotkey); (b) get the PRD to downgrade FR-9a from "reachable" to "documented as not reachable while locked." Recommend (a) — costs one comparison per event and restores a real OS-level escape. |
| REV-8 | Major | UI-3 / FSM-5 | Hold-to-unlock implementation is left as "use HoldButton NSViewRepresentable; fall back to SwiftUI gestures if NSViewRepresentable proves problematic." This is exactly the kind of "engineer decides at implementation time" ambiguity the spec is supposed to eliminate. SwiftUI's `LongPressGesture` does not expose intermediate progress; `DragGesture(minimumDistance: 0)` + a timer is the workaround but has well-known issues coexisting with button hit-testing. | Pick HoldButton and write the contract: subclass `NSControl`, override `mouseDown`/`mouseUp`/`mouseExited`, expose a `@Published var progress: Double`, drive a `CADisplayLink` (or `CVDisplayLink`) for 60 Hz updates. Define behavior when the cursor exits the button frame mid-hold (cancel? continue?). Specify it once. |
| REV-9 | Major | TAP-6 / DATA-1 | The flow from `PreferencesStore` to `LockController`'s lock-protected `HotkeyBinding` is underspecified. DATA-1 says the store "publishes a snapshot to LockController"; TAP-6 says the binding "is held in an `OSAllocatedUnfairLock` on the controller." Who calls whom? Combine subscription? Direct method call? Where does the FR-8a "only writable when unlocked" invariant get enforced? | Define explicitly: `PreferencesStore.unlockHotkey` is `@Published`; `LockController.init` takes the store and subscribes via Combine on a serial queue; writes are gated by `LockStateMachine.isUnlocked` (assert + early return); the controller writes through the `OSAllocatedUnfairLock` inside the subscription handler. One paragraph in TAP-6 fixes this. |
| REV-10 | Major | PERM-4 / PERM-5 | macOS frequently requires the app to be **re-launched** after first granting Accessibility before `CGEventTap` creation actually succeeds, even though `AXIsProcessTrusted` returns `true`. The spec's polling loop will flip the state to `unlockedReady`, the user will click Lock, and tap creation will fail silently or partially. Not addressed. | Add: after the silent permission check returns granted, attempt a probe `CGEvent.tapCreate(...)` with a no-op callback and immediately invalidate it. If creation fails, treat it as `unlockedNeedsPermission` and show a "Restart KeyboardLock to finish setup" banner with a "Restart Now" button (calls `NSApp.terminate` + `LSOpenApplicationAtURL` via a 1-shot launch agent OR simply asks the user to re-open it). |
| REV-11 | Major | UI-5 + EDGE-1 | When the watchdog forces an unlock from a background queue while main is hung, the floating panel keeps showing "Keyboard is locked," the menu bar icon keeps pulsing, and the user has no visual signal that they can now type. The tap is gone but the UI lies. | Required pair to REV-1: the watchdog must (a) hide the floating panel via `NSApp.windows[...].orderOut(nil)` or by closing the `NSPanel` directly (AppKit window operations are documented thread-safe for orderOut/close); (b) flip the menu bar icon by writing to a non-`@Published` atomic flag the menu's `NSStatusItem` timer reads. State this contract in EDGE-1. |
| REV-12 | Major | TAP-4 modifier mask | The mask includes `.maskSecondaryFn`. If a user records ⌃⌥⌘L while not pressing fn (the normal case) and then accidentally rests a finger on the fn key during cleanup, the matcher's `flags == binding.requiredFlags` comparison fails and unlock silently no-ops. Worse, on MacBooks with the "globe" key remapped to fn, this is easy to trigger. | Either (a) exclude `.maskSecondaryFn` from both the binding and the comparison (recommended — fn is rarely an intentional modifier) or (b) document in the recorder UI that fn-state is part of the binding. State the choice. |
| REV-13 | Major | TEST-I1 / TEST-I2 | Synthetic events posted via `CGEvent.post(tap: .cgSessionEventTap)` from a test process are injected **above** `kCGHIDEventTap`, so the production tap will not see them. The test would be exercising a different code path than production. | Post via `CGEvent.post(tap: .cghidEventTap)` (which requires the test process itself to have Accessibility) and verify drop by observing whether a separate listener tap installed at `.cgSessionEventTap` sees the event. Document the Accessibility-grant prerequisite for these tests explicitly. |
| REV-14 | Major | SPEC-PRDGAP2 | The spec flags this but defers ("Confirm with PM"). Cannot ship with this ambiguity: if a user disables the menu bar item in prefs, FR-13's "menu bar SHALL reflect state at all times" is unsatisfied, and the user loses the FR-15 pulsing-while-locked indicator. | Make a decision in the spec: when "Show menu bar icon" is disabled, the locked-state visual contract reverts to the always-on-top floating panel only. Note that disabling the menu bar item is therefore subtractive of one safety signal. Either accept this trade-off in writing or remove the preference. |
| REV-15 | Minor | DATA-1 | `prefs.hasCompletedFirstRun: Bool` is declared but never referenced — the first-run UI is driven entirely by permission state. Either dead code or hidden requirement. | Delete the key, or describe what it gates. |
| REV-16 | Minor | UI-6 | "Pre-lock countdown" is a segmented control (Off / 1s / 3s / 5s); PRD FR-2 specifies "skippable via a checkbox." Functionally equivalent but a deviation. | Either align with the PRD checkbox (countdown on/off, fixed 3s when on) or note the deviation as a deliberate UX improvement and have the PRD updated. The segmented control is the better design — recommend the PRD edit. |
| REV-17 | Minor | UI-3 / UI-5 | NSPanel with `[.titled, .nonactivatingPanel]` is correct for not stealing focus, but the spec does not address whether mouse events to the HoldButton inside a non-activating panel route correctly to SwiftUI's gesture system. There are documented quirks here. | Prototype this exact configuration first, before any other UI work. If `mouseDown` on the non-activating panel content doesn't fire the SwiftUI gesture, fall back to a regular `NSWindow` at floating level with `becomesKey = false` overridden. |
| REV-18 | Minor | SPEC-Q1 (single-instance) | Open question left dangling. Macos's default behavior (relaunch brings existing instance forward) is fine for normal launches, but `open -n /Applications/KeyboardLock.app` would spawn a second instance, and a second tap install could fight the first. | Decide: enforce single-instance via `NSWorkspace.runningApplications` check in `applicationDidFinishLaunching` and call `NSApp.terminate(nil)` if a sibling is found. One-line decision. |
| REV-19 | Minor | EDGE-2 | `NSWorkspace.willSleepNotification` is posted only ~1 second before sleep; the spec assumes there's enough time to tear down the tap and release the assertion. Usually there is, but it's not guaranteed. | Document the assumption. On wake, additionally re-check the tap state and force `unlockedReady` defensively even if the will-sleep handler didn't complete. |
| REV-20 | Minor | UI-7 | "Countdown" state has color cue "green→amber gradient on ring" but FR-10 mandates that transitional states are visually distinct from both Unlocked and Locked. Gradient between the two endpoint colors is technically distinct but visually weak. | Add an explicit transitional accent (e.g., system blue) and a "Locking in N…" headline so it doesn't read as "almost locked / almost unlocked." |
| REV-21 | Nit | §1 intro table vs BUILD-5 | "Third-party deps: None required in v1. Sparkle considered." Pick one in the table once REV-4 is resolved. | Edit the table after REV-4 decision. |
| REV-22 | Nit | UI-5 | "Custom asset for v1.1" decision deferred (SPEC-Q5). Acceptable; using `keyboard.badge.ellipsis` or composite SF Symbol for v1 is fine but worth confirming the composite renders crisply at 16/32/64 pt menu bar sizes. | Snapshot test the icon at all retina factors. |

## What the spec gets right (briefly)

- Tap placement at `kCGHIDEventTap` with `.headInsertEventTap` is the correct choice and correctly justified.
- Honest enumeration of TAP-8 / L-1…L-6 limits — secure input, hardware power button, Touch ID, firmware-handled keys. No false promises.
- Separation of LockController onto its own thread with a dedicated CFRunLoop is the right architecture.
- Hotkey-matcher value type + `OSAllocatedUnfairLock` snapshot is a sound lock-free-on-hot-path design (modulo REV-9's missing wiring).
- The choice to never persist `LockState` across launches (DATA-2) is correct — booting into a locked state after a crash would be a foot-gun.
- Traceability table is genuinely useful and mostly accurate.

## Must-fix before coding

Only blockers and majors. Resolve all of these and the spec is implementable.

1. **REV-1** — redesign watchdog ↔ state-machine concurrency so background mutations don't touch `@Published` / `@MainActor`.
2. **REV-2** — remove or harden the menu-bar "Unlock Keyboard" item; do not bypass the FR-7 safeguard.
3. **REV-3** — define `unlockingDrain` sub-state and keep the tap installed for a debounce window after a matched hotkey.
4. **REV-4** — resolve the Sparkle vs. "no network calls" contradiction. Recommended: defer Sparkle to v1.1.
5. **REV-5** — replace TEST-M6 with a test that actually starves the main queue, and add the watchdog clock-injection seam.
6. **REV-6** — specify a zero-leak re-arm contract for `tapDisabledByTimeout`, including UI behavior during the gap.
7. **REV-7** — either allow ⌥⌘Esc through the tap, or amend PRD FR-9a.
8. **REV-8** — commit to `HoldButton`-as-`NSControl`; specify cursor-exit behavior; remove the "fall back to SwiftUI" hedge.
9. **REV-9** — write the explicit PreferencesStore → LockController binding-update flow.
10. **REV-10** — handle the "Accessibility granted but tap creation still fails until relaunch" reality with a probe-tap and a relaunch prompt.
11. **REV-11** — pair the watchdog teardown with explicit UI dismissal so the panel doesn't lie after a forced unlock.
12. **REV-12** — decide whether `fn` is part of the hotkey binding; recommend excluding it.
13. **REV-13** — fix the integration-test injection point so tests exercise the same tap level as production.
14. **REV-14** — make a decision on FR-13 vs. the "hide menu bar" preference; do not defer.
